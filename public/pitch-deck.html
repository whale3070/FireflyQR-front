Why On-Chain

For long-lifecycle goods (notebooks, phones, luxury items), traditional databases fail: data can be altered, companies can shut down, and traceability becomes meaningless. On-chain data never disappears. Every mint, transfer, and secondary activation is verifiable, permanent, and traceable to the original issuer.

  • ❌Centralized data can be manipulated — brands self-report; no audit trail
  • ❌When companies vanish, traceability data is gone — no recourse for consumers
  • ❌No trusted record for second-hand flow — resale market is opaque and risky
Why Database-Only (No Chain) Can’t Solve This

Cross-border: Buyer and seller don’t trust the same central party. A database run by one side (or one vendor) is not trusted by the other — data can be changed, and there is no shared, immutable proof. Trust between cross-border parties cannot be built on a single centralized database.

Second-hand: The two sides have no tamper-proof digital ID for the product: when it was bought, whether it was repaired, whether it’s genuine. A central DB is either controlled by one side (the other won’t trust it), or by a third party that can alter data or shut down. There is no neutral, verifiable record that both buyer and seller can rely on.

On-chain: the same record is visible and unchangeable for everyone. The NFT is the product’s portable, unforgeable ID — purchase time, transfers, and (if recorded) repair history live on a ledger neither party controls. That’s why we use the chain.

Financing Pain Point We Solve

Cross-border SMEs face hard and expensive financing, and funding chains break easily across the full trade lifecycle (pre-production, logistics, settlement). Banks and investors lack verifiable proof of real sales and real users.

With FireflyQR, on-chain real sales, NFT ownership, and real user data become auditable collateral. Merchants can use this proof to borrow from investors and banks — borrow and repay on demand, without locking into rigid loan terms. We address: SME financing difficulty, high financing cost, and fragile trade funding chains.

Break Channel Monopoly — Direct Reach & Private Domain

Today, platforms monopolize distribution: JD, Taobao, and similar marketplaces control the channel. Producers get thin margins; middlemen capture most of the value. Brands don’t own the customer relationship — the platform does.

With this solution, every scan brings the consumer into the brand’s private domain (e.g. community, follow-up, repurchase flow). You get higher repurchase and direct reach — no need to rely on JD, Taobao, or other centralized platforms to touch the end user. Producers can retain margin and own the relationship; the middleman layer is no longer the only path to the customer.

Solution: Three Pillars

1. Anti-counterfeiting — One code per item; each NFT is minted from the merchant’s official contract. Provenance is verifiable: the NFT’s contract address proves it came from the brand, not a counterfeiter.

2. Traceability & anti-diversion — On every scan we capture the user’s IP, salt it, and store it in the database. You get regional and channel analytics and can detect diversion (e.g. goods intended for region A appearing in region B).

3. Financing — Real scan volume, scattered timestamps, and geographic distribution prove genuine end-user activity. Banks and investors can treat this as auditable proof for borrow-and-repay financing.

Why the QR Code Resists Copying

The code is only visible after unsealing the package, so it can’t be copied at scale without opening the product. When the user claims the reward (e.g. red packet), a post-sale countdown starts and the claim is recorded (time, location, amount, SKU).

If a code is batch-copied and shared, anyone who scans it later can see when, where, and how much was claimed, plus the SKU. If that doesn’t match the product they’re holding (wrong SKU, already claimed, wrong region), the copied code effectively tells the customer: this is counterfeit. Copying the QR doesn’t help fakes — it exposes them.

How It Works

Each physical good is bound to a hash code and an on-chain NFT. When a customer scans a QR or redeems a code, a verifiable claim event is generated.

Pipeline
  • 1📦 Real-world scan or redeem (QR / link)
  • 2🔗 Backend validates code (one-code-one-claim) and captures scan IP — salted and stored in DB for traceability and anti-diversion
  • 3💰 Gasless mint via relayer — user enters address, no wallet needed
  • 4🔄 Secondary activation — second-hand buyer pays 1.1 native token to transfer NFT ownership
Traceability & Anti-Diversion

On each scan we record the user’s IP address, hash it with salt, and store it in the database (privacy-safe, no raw IP retained). This gives merchants:

  • ✔Regional analytics — See where scans happen (by region/country) to align with sales channels
  • ✔Diversion detection — Spot when goods meant for one region or channel are activated elsewhere
  • ✔Financing proof — Scattered timestamps + geographic spread make fake volume easy to detect; real data supports lending
NFT Deposit Pool & Authenticity ID

The FireflyQR RWA system has an NFT deposit pool that only increases, never decreases. The design is inspired by the mature German bottle deposit (Pfand) model: deposits create incentive for someone to maintain and return the asset, so the NFT stays tied to the physical good and is properly “cared for” along the chain.

Second-hand buyers use the NFT to verify genuine origin — no need to trust the seller’s word. The NFT acts as the product’s authenticity ID: one scan, one record, one source of truth. That simplifies verification for high-ticket goods (luxury, jewelry) so radically that dedicated LV authentication and jewelry authentication services become unnecessary: the NFT is the proof. No third-party “expert” layer required.

Web2-Friendly UX

Users do not need wallets or native tokens for first claim. Gas is abstracted via relayers. Secondary activation requires MetaMask (or compatible wallet) to send 1.1 PAS to the contract.
